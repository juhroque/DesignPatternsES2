Aprovação de Tarefas (Chain of Responsibility)

Fluxo de Aprovação

Tarefas concluídas entram no fluxo de aprovação, passando por três níveis consecutivos:

  1. Viewer: Validação inicial
  2. Editor: Revisão técnica
  3. Admin: Aprovação final

Seu nível de acesso determina quais aprovações pode realizar.

Explicação dos Design Patterns Implementados

Objetivo: Fornecer um substituto ou espaço reservado para outro objeto para controlar o acesso a ele.

Implementação: Usamos um Proxy de Controle de Acesso que atua como intermediário entre o controlador UI e a lista de tarefas, validando as permissões do usuário antes de permitir operações como criar, editar ou excluir tarefas.

Benefício: Separa a lógica de controle de acesso da lógica de negócios, permitindo adicionar regras de segurança sem modificar as classes originais.

Sobre a demonstração: Esta implementação simula diferentes níveis de acesso para fins didáticos. Em um sistema real, o usuário não poderia selecionar seu próprio nível de acesso.

// Exemplo simplificado do Proxy
class AccessControlProxy {
    constructor(taskList) {
        this.taskList = taskList;
        this.accessManager = new AccessManager();
    }
    
    deleteTask(taskId) {
        // Verificar permissão antes de executar a operação
        if (this.accessManager.hasPermission('delete')) {
            const index = this.taskList.findIndex(task => task.getId() === taskId);
            if (index !== -1) {
                this.taskList.splice(index, 1);
                return true;
            }
        } else {
            console.log('Permissão negada');
            return false;
        }
    }
}

Implementação: Criamos um gerenciador de configurações (ConfigManager) que mantém uma única instância para toda a aplicação, gerenciando as preferências do usuário.

Benefício: Centraliza o acesso às configurações da aplicação, evitando duplicação de dados e garantindo consistência.

Objetivo: Criar objetos sem especificar a classe exata do objeto a ser criado.

Implementação: Usamos o Factory Method para criar diferentes tipos de tarefas (Pessoal, Trabalho, Estudo) sem expor a lógica de criação ao usuário.

Benefício: Facilita a adição de novos tipos de tarefas sem modificar o código que usa as tarefas.

// Exemplo simplificado
class TaskFactory {
    createTask(type, title, description) {
        switch(type) {
            case 'personal': return new PersonalTask(title, description);
            case 'work': return new WorkTask(title, description);
            case 'study': return new StudyTask(title, description);
        }
    }
}

Objetivo: Adicionar responsabilidades a objetos dinamicamente sem alterar sua estrutura.

Implementação: Usamos Decorators para adicionar comportamentos como Alta Prioridade, Etiquetas Coloridas e Datas de Vencimento às tarefas existentes.

Benefício: Permite combinar comportamentos de forma flexível sem criar uma explosão de subclasses.

Objetivo: Definir uma dependência um-para-muitos entre objetos para que quando um objeto mude de estado, todos os seus dependentes sejam notificados.

Implementação: Usamos o Observer para notificar diferentes canais (Tela, Email simulado, Log) quando o status de uma tarefa muda.

Benefício: Desacopla o objeto observado dos observadores, permitindo adicionar novos observadores sem modificar o objeto.

// Exemplo simplificado
class TaskSubject {
    constructor() {
        this.observers = [];
    }
    
    addObserver(observer) {
        this.observers.push(observer);
    }
    
    notifyObservers(task, status) {
        this.observers.forEach(observer => {
            observer.update(task, status);
        });
    }
}

Objetivo: Passar uma solicitação ao longo de uma cadeia de manipuladores, onde cada um decide se processa a solicitação ou a passa para o próximo na cadeia.

Implementação: Usamos o Chain of Responsibility para implementar um fluxo de aprovação de tarefas, onde cada nível de acesso (Viewer, Editor, Admin) representa um manipulador na cadeia.

Benefício: Desacopla o emissor de uma solicitação dos receptores e permite que múltiplos objetos tenham a chance de processar a solicitação.

// Exemplo simplificado
class Handler {
    constructor() {
        this.nextHandler = null;
    }
    
    setNext(handler) {
        this.nextHandler = handler;
        return handler;
    }
    
    process(request) {
        if (this.nextHandler) {
            return this.nextHandler.process(request);
        }
        return false;
    }
}
